# Changelog

All notable changes to this project are documented here. Format follows
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versioning follows
[Semantic Versioning](https://semver.org/) (`version` in `pyproject.toml` is
the source of truth — bump it by hand and tag the commit `vX.Y.Z`).

## [Unreleased]

### Added
- `report/publish.py`: copies a race's sanitized report and the running summary (`reports/<race_date>.md`, `reports/SUMMARY.md`) into a local fsrc-tech checkout's `docs/race-services/reports/` folder (`python -m chip_timing_analysis.report.publish <race_date> <fsrc_tech_dir>`, or the new "Chip Timing: Publish Report to fsrc-tech" VS Code task). Automates only the file-copy steps of the publishing workflow — adding a fsrc-tech `CHANGELOG.md` entry and committing/pushing there stays a manual, deliberate step, per Lou's call not to run those proactively.

### Fixed
- `.vscode/tasks.json`'s optional `promptString` inputs (`raceName`/`raceDate`/`gunTime`/`note`) no longer silently abort the whole task when left blank and confirmed with Enter — a `promptString` whose `default` is exactly `""` can resolve as *cancelled* rather than as an empty string in VS Code's own input handling. Now default to a single space `" "` instead; `generate.py`'s blank-detection strips whitespace before checking rather than comparing to exact `""`.
- The `distance` picker's auto-detect option also had this problem one level deeper: its value of `""` vanished entirely from the assembled shell command line (rather than being passed as an empty argument), collapsing `--distance` and the following `--gun-time` flag together and breaking argument parsing. Same fix — the option's value is now `" "` too.

## [0.2.0] - 2026-07-12

### Added
- Per-mat raw-read diagnostics (`analysis/mat_reliability.py`), surfaced in the detailed report only: flags an antenna reading weakly (low signal strength) or being missed almost entirely relative to its immediate neighbors, a possible sign of a cabling/power problem worth a physical check.
- Running summary table (`reports/SUMMARY.md`)'s Date column now links each row to that race's sanitized report; documented the manual workflow for publishing sanitized reports/summary into the fsrc-tech wiki (`docs/race-services/reports/`).
- Sanitized per-race report (`reports/<race_date>.md`) now opens with a `title` front-matter block so its wiki nav entry shows the race date rather than a generic heading. Running summary retitled "Timing Summary", now lists most recent race first, drops the "No Chip" column (folded into the Notes cell only when nonzero), and truncates the Notes cell to keep mkdocs-material's table from wrapping long text into tall rows.
- `parsers/rdgo_export.py`: parses RaceDay Scoring's full database export zip (`<race name>.json` + `filterer.data`) and is now the normal ingestion path in `build_race_report()` when a zip is present in `race_dir` — supersedes `bib-chip.csv`/`tm-data.csv`/`backup-time-selected.csv`/`time-machine-used.csv` entirely, and `gun_time` becomes optional (read from the export). Falls back to the original CSV-based ingestion when no zip is present (e.g. Indy 5000). Validated end-to-end against the Parkway Panda 5K: reproduces the previously-validated numbers (12 no-chip, 0 zero-read, 5 missed-start, 1.5% missed) via the new path. See CLAUDE.md and `~/.claude/plans/graceful-honking-sky.md` for the full writeup.
- `RaceReport.network_interruption_suspects`: when both an rdgo export and `f_*.log` are present, cross-checks every rdgo-reported miss against the local Trident log independently (the export only reflects what reached RDS live) — same failure mode behind Indy's 576/591 case. Not yet surfaced in `render.py`.
- `build_race_report()`/`generate()` take an `exclude_bibs` parameter — bibs Lou confirms should not count as a participant at all (e.g. a chip cross-reference in RDS that's a data-entry error, never actually issued to that participant), excluded from every count entirely. Distinct from `known_drops`, which still counts the bib as a real participant who started but didn't finish.
- `parsers/rdgo_export.py` now supports a combined multi-distance race (e.g. Wild Trail 5K & 10K) whose zip export has more than one RDS "scored event" sharing one timing setup — `RdgoExport.scored_events` (plural) replaces the old single `scored_event`, and `bib_gap_factors()` resolves each bib's own distance-specific Gap Factor (`scored_event.min_finish_time`) rather than one shared value read off `timing_location.gap_factors` (which happened to coincide with the 5K's value on every single-distance race checked so far — Indy 5000, Panda 5K — masking the distinction until a combined race exposed it). `report/build.py`'s CSV-based cross-checks (pre-gun-grace backfill, `network_interruption_suspects`) now run per gap-factor group (`_csv_classify_grouped()`) instead of applying one race-wide gap factor. Validated against Wild Trail 5K & 10K (87 participants, 1 missed start, 1 finish miss recovered via backup, no weak mats) and re-verified Indy 5000/Panda 5K reproduce their exact previously-validated numbers unchanged.
- `report/generate.py`'s CLI is now `argparse`-based and exposes every `generate()`/`build_race_report()` parameter (`--n-registered`/`--n-starters`/`--n-finishers`/`--detail-note`/`--known-drop`/`--exclude-bib`, previously Python-only). `race_name`/`race_date`/`distance`/`gun_time` are now optional flags rather than required positionals — when `race_dir` has an RDS export zip, any left unset are derived from it (`race_name` from `race["name"]`, `race_date` from `race["start_date"]`, `distance` from the export's own `integration_events` via the new `rdgo_export.distance_label()` — preferred over reverse-mapping Gap Factor, which is a user-adjustable timing setting, not a fixed distance encoding).
- `.vscode/tasks.json`: "Chip Timing: Generate Race Report" VS Code task runs the CLI above via Command Palette → "Run Task". For a zip-based race, `race_dir` is the only field that must actually be filled in — every other prompt can be left blank and is auto-derived.

### Fixed
- `RaceReport.genuine_finish_miss_bibs()` now excludes confirmed drops (`known_drops`) from the "chips not read at finish" headline count and `pct_read_opportunities_missed()` — a confirmed drop (started, did not finish) was never going to have a finish read, so it isn't a chip-reading miss. Previously counted as a subset of "unrecovered finish misses"; now reported as a fully separate line/section. On the Indy 5000 this changed the headline from 4 to 3 finish misses (2.3% → 1.8% missed).
- `parsers/backup_time_selected.py` no longer crashes when "Chip Start Time of Day" is blank — a bib with no chip assigned at all (backup was its only possible source) legitimately has no chip-start value; now parsed as `NaT` instead of failing the whole file.
- `report/render.py`'s `_reconcile_backup()` and the detailed report's "Backup timing used" table no longer mislabel a no-chip-assigned bib as a "network interruption" case — both are backup-used and both are absent from `finish_miss_bibs()`, but for different, already-reported reasons. Found on the Parkway Panda 5K, the first race in this repo where a no-chip bib also appeared in the backup report; Indy never exercised this path.
- `analysis/start_finish.py`'s `classify_start_finish()` now accepts a `pre_gun_grace` (default 1s) and includes reads from `gun_time - pre_gun_grace` in the start window — a few front-of-field runners' chips cross a fraction of a second before the automatic GUNTIME marker registers, and were previously misclassified as a missed start. Found on the Parkway Panda 5K (3 of 8 "missed start" bibs had a clean LS read 0.04-0.30s before gun_time); reproduces Indy's validated 1 missed start (641) unchanged.
- `report/build.py`'s rdgo-export ingestion path no longer mislabels a backup-recovered finish (chip missed it, Time Machine correctly substituted) as a "network interruption" — `read_data`'s `occurrence` is present whenever *either* stream provided a value, so `missed_finish_only`/`missed_start_only` now use a chip-only view (`is_chip_device()`-filtered) for miss *detection* while still rendering the real, backup-inclusive times. Restored the previously-validated Panda 5K numbers (missed_finish_only 148/267/277/303/331/333; genuine misses 148/277/331 recovered via backup).
- `parsers/rdgo_export.py`'s `bib_chip()`/`known_drops()`/`classify_start_finish()` now restrict to the target scored event's own participants (`_in_scored_event()`) — `entity_map` covers every registrant under the race umbrella, not just the one scored event being analyzed.
- `parsers/rdgo_export.py`'s `known_drops()` now excludes participants who never started (`rdgo_dns` True) — RDS sets `rdgo_dnf` on every non-finisher as routine bookkeeping, including no-shows, which had inflated Panda 5K's auto-detected drop list to ~42 bibs when only 4 were real (267/303/333/375, all `rdgo_dns=False`).
- `report/build.py`'s rdgo-export ingestion path no longer regresses the `pre_gun_grace` fix — RDS's own `read_data` has no equivalent grace period, so bibs whose only start read fell just before gun_time (180/314/349 on Panda 5K) reappeared as "missed start" until wiring this up as the primary path. Now backfilled from the CSV-based classification specifically for the pre-gun-grace case before treating a missing rdgo start as real.
- `RaceReport.pct_read_opportunities_missed()` no longer counts a no-chip-assigned bib as "2 missed opportunities" — there was no chip, so there was no chip-read opportunity to miss (per Lou). `zero_reads` still counts as 2 missed (a chip *was* assigned and genuinely never worked). Denominator unchanged. On Panda 5K this dropped the headline from 6.1% to 1.5%; Indy unaffected (never had a no-chip bib).
- `report/render.py`'s `append_summary_row()` now inserts by race date descending instead of blindly prepending, and replaces an existing row for the same race instead of duplicating it. The old behavior silently broke "most recent first" the moment a race got (re-)generated out of chronological order — exactly what happened this session re-running the Parkway Panda 5K (2026-05-25) after the Indy 5000 (2026-07-04) was already in the table, and every re-run left a stale duplicate row that had to be removed by hand.

## [0.1.0] - 2026-07-09

Initial release. Parsers for Trident raw/filtered read logs, bib/chip
assignments, tmtility data export, barcode-scanner log, and RDS backup-time-
selected report. Start/finish classification against the RDS gun-time/gap-
factor theory, finish-to-bib-scan queue lag, and finish-line density
analysis. Per-race report builder and renderer (detailed + sanitized +
running summary), validated end-to-end against the Indy 5000 race.
