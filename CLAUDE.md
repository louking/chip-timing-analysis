# chip-timing-analysis

Offline analysis tool for FSRC race chip timing data (Trident Time Machine reader). Parses raw reads, tmtility logs, and tmtility data exports into pandas DataFrames to compute things like missed start/finish reads, finish-to-bib-scan queue lag, per-reader/mat read counts, and read gaps — eventually a per-race report.

## Relationship to tm-csv-connector

Deliberately a separate repo, not tooling inside `tm-csv-connector`. `tm-csv-connector` is a deployed live-ingestion service (Flask/SQLAlchemy, `trident2db`/`tridentread2obj` in `tm_csv_connector/trident.py`) coupled to its DB models (`ChipBib`, `ChipRead`, `Race`) and its own docker/fabric release pipeline. This repo is offline/exploratory pandas analysis with different dependencies and a different release cadence — bolting it onto the connector risks bloating that deployment.

To avoid the two repos' understanding of the Trident wire format drifting apart, this repo's raw-read parser ports the byte offsets from `tm-csv-connector/app/src/tm_csv_connector/trident.py` (`tridentread2obj`/`tridentmarker2obj`) rather than re-deriving them from the spec doc.

## Trident raw read format (verified against tm-csv-connector and a real race export)

Line-oriented, prefix at `line[0:2]`. Only `aa`/`ab`-prefixed lines are read records — everything else (device banner text, JSON telemetry — see below) must be skipped, not just assumed absent.
- `aa` = chip read record, 40 chars: `reader_id`=line[2], `receiver_id`=line[3], `tag_id`=line[4:16], `counter`=line[16:20], `date`=line[20:26] (YYMMDD), time=line[26:28]h/line[28:30]m/line[30:32]s + hex hundredths line[32:34], `rtype`=line[36:38] (FS/LS/BS/RR), `rssi`=line[38:40] hex (optional — often absent on non-RR records).
- `ab` = marker/gun-time record, **30 chars** (not 40 — different width and layout from `aa`, not a chip read): `reader_id`=line[2], `date`=line[8:14], time=line[16:18]h/[18:20]m/[20:22]s + hex hundredths line[22:24]. The first `ab` GUNTIME record seen for a race is the actual gun time. Seen in practice as two markers a few hundredths apart (multiple reader units logging the same trigger).
- Raw files also end with a few free-text banner lines (`System version ...`, `Hardware version ...`, `Hardware MAC address ...`) — not fixed-width, must be filtered out by the `aa`/`ab` prefix check, not by line length.

Trident export file naming convention (seen in real exports): `r_YYMMDD.log` = raw, all chip reads (`RR` type). `f_YYMMDD.log` = filtered/best-seen (`BS`/`LS` types) — **interleaves JSON telemetry lines** (`{"uptime":...,"gps":...,"battery":...}`) between record lines, so the `aa`/`ab`-prefix filter is required here too. `s_YYMMDD.log` = human-readable periodic statistics blocks, not read records.

`tm-data.csv`: no header, columns `seq,bib,time` (time = `HH:MM:SS.hh`). **This is the Time Machine backup stream's data, not RDS's final scored result** — RDS "ignores backup-stream reads entirely once any main-stream (Trident) read exists for a participant" (fsrc-tech `race-services/timing.md`), so this file's time is only authoritative for bibs where the Trident chip was never read at the finish. For everyone else it's just the manual backup timer's button-press, incidentally close to (but not the source of) the real chip-timed finish.

`bib-chip.csv`: header `bib,chip` — `chip` is the same 12-char hex tag ID as `tag_id` in raw reads, but **uppercase** in this file vs. lowercase in the raw log (e.g. `00000C08C1D5` vs `00000c08c225`) — normalize case before joining.

## How RaceDay Scoring determines start/finish reads (validated against Indy 5000 data)

Authoritative sources: fsrc-tech `race-services/timing.md`, and Lou's RDS setup checklist (Google Sheet, id `1GcDuB508jKu_kdMXZ8VhzIaZk6LE2l4qIxwpstLUKIc` — readable via the Google Drive MCP tool `read_file_content`, not a plain web fetch).

Don't assume `mat_id` maps to a fixed station (e.g. "mat 1 = start"). For chip races, RDS's Start and Finish Timing Locations both point to the same combined "Start/Finish" location — the same physical antenna set (mat_ids 1-8 on the Indy 5000) reads both crossings. Classify crossing events by **time of day**, not `mat_id`:

1. **Start**: reads within `[gun_time - pre_gun_grace, gun_time + Max Chip Start Time Offset]` (2 min in the current checklist; `pre_gun_grace` defaults to 1s — see the Parkway Panda 5K note below). Use the **LS** (Last Seen) read.
2. **Finish**: reads at or after `gun_time + Min Elapsed Finish Time Allowed` (a.k.a. Gap Factor for occurrence 1) — **race-distance-specific**: 14 min (5K), 30 min (10K), 50 min (10M). Within that window, split into occurrences wherever the gap between consecutive reads for the tag is ≥ the same Gap Factor value, and take the **BS** (Best Seen) read of the first occurrence (RDS's "Default Finish Occurrence: 2", counting the start-window pass as occurrence 1). Trident's own BS algorithm takes ~0.5s to settle, per Trident support.
3. Reads before the `GUNTIME` marker, and any reads for a tag after its finish occurrence, are noise and should be ignored (confirmed by Lou).

Validated on the Indy 5000 (5K, 110 known bibs) using this exact method (2 min start window, 14 min finish-window start / Gap Factor): 107/110 bibs' computed finish (occurrence-1-in-finish-window BS) matched `tm-data.csv` finish_time within -1.4s..+1.2s (median +0.42s — consistent with the ~0.5s BS settle time). The 3 that didn't match had **zero** Trident reads in the finish window at all (bibs 598, 606, 630) — these are genuine missed finish reads, exactly the case where RDS would fall back to the `tm-data.csv` (TM backup) time. 1 bib (641) had no read in the start window — a genuine missed start read.

**Known limitation, explained by Lou:** on race day RDS actually used a backup stream for 5 bibs — 598, 606, 630 (the zero-read cases above, genuine missed reads) **plus 576 and 591**, which `missed_finish_reads()` can't detect from the log files alone. Reason: the local Trident log files (`r_*.log`/`f_*.log`) reflect what the **reader device itself** logged, not what reached RDS live — a network dropout during the race can mean a read exists in the device log (576 and 591 both have a clean BS read matching `tm-data.csv` within ~0.6s) but never arrived at RDS's live main stream, so RDS correctly fell back to backup *at the time*. RDS has a secondary "Trident File" backup stream mechanism to recover from exactly this (re-import the uploaded log post-race), but that step wasn't done on race day for Indy — so every backup case that day happened to be TM specifically; don't assume that holds for other races (see `backup_time_selected.py` below). Net effect: **`missed_finish_reads()` finds genuine antenna/positioning misses (zero reads in the log), not live-network-transmission misses** — the two are different failure modes with different fixes, and only the log-based one is detectable from raw/filtered reads alone.

**Cross-referencing finish-line video + RDS's own report to pin down the outage window:** the finish-line camera (see fsrc-tech `timing.md`) is carried on the same local network as the Trident live feed, so a gap in the recorded video is independent evidence of a network outage window. Convert the video's displayed race-clock time (mm:ss.hh, from the manually-started display clock, not the camera's own timestamp overlay) to wall-clock via `gun_time + elapsed`, then filter `classify_start_finish` output to that window to see who finished during it. On Indy, a video gap of 35:09.93-38:52.85 elapsed (09:05:12-09:08:55 wall-clock) contains 12 finishers in the log. Cross-referencing against RDS's "Backup Time Selected" report (ground truth — exactly bibs 576/591/598/606/630, confirmed, no others) narrows this considerably: 614 and 616 (09:05:51/:47, each with an unusually high 8 finish-window reads) did *not* need backup despite finishing right at the start of the video gap, and 615 (09:07:04) also didn't need backup despite finishing soon after 576 (09:06:51, did need backup) — so the **actual live-network outage was ~09:05:51 to ~09:06:52, about one minute**, well inside but much shorter than the 3.5-minute video gap (there's likely a beat of human reaction time between the automatic `GUNTIME` marker and someone manually starting the display clock, so `gun_time + elapsed` from video is only approximate).

## tmtility app logs (barcode-scanner.log / tm-reader.log)

These are the `tm-csv-connector` service's own operational logs (standard Python `logging` format), not chip data — websocket connections, `raceid`/`open`/`close` opcodes, exceptions. They are long-running app logs spanning far beyond any one race (can cover months) — **shared across races, not race-scoped**, so they live once at `data/logs/`, not duplicated into each race directory. Always filter to the race's actual date/time window first (`parse_barcode_scans`'s `start`/`end` params).

`barcode-scanner.log` contains the useful signal: lines like `barcode scanner data received: <bib>` immediately followed by `sending to backend: {'opcode': 'scannedbib', 'raceid': ..., 'bibno': ...}`, timestamped to the millisecond. These correlate closely with `tm-data.csv` finish times (e.g. bib scanned a few seconds after its chip finish time) — this is the primary source for the finish-to-bib-scan queue lag metric.

## Project setup

Python venv at `.venv/` (not committed), deps in `requirements.txt` (currently just `pandas`). Package lives under `src/chip_timing_analysis/`, installed editable (`pyproject.toml`, setuptools, `pip install -e .`) so cross-module imports (e.g. `from chip_timing_analysis.parsers.trident_raw import ...`) work regardless of cwd — run modules via `-m chip_timing_analysis...` (no `src.` prefix; earlier docs/history may show the old `-m src.chip_timing_analysis...` form from before the editable install was added, don't copy that), e.g.:

```
.venv/Scripts/python.exe -m chip_timing_analysis.parsers.trident_raw "data/2026-07-04-indy-5000/r_260704.log"
```

`parsers/trident_raw.py` has `parse_trident_raw(file_path) -> pd.DataFrame` (columns: tag_id, reader_id, mat_id, timestamp, read_type, signal_strength, read_count; `aa` chip-read records only, `ab`/banner/telemetry lines skipped) and `sanity_check(df)` (record count, date range, unique tags, read-type distribution) — validated against `r_260704.log` (6397/6397 records, 100% RR, matches independent count). Since it filters purely on the `aa` prefix, it also works unmodified on `f_*.log` (confirmed: 674 records, 337 BS + 337 LS, JSON telemetry correctly skipped) — no separate filtered-file parser needed.

**Raw vs. filtered, which to use for which metric:** `f_*.log` (BS/LS) collapses each tag's mat crossing to one representative read, so it's the right source for anything about *crossing events* — missed start/finish reads, finish-to-bib-scan queue lag, finish-line density. `r_*.log` (RR, every individual antenna read) is required for *read-density/positioning diagnostics* — per-mat read counts and read-gap analysis depend on how many times and how consistently a tag was picked up while in the antenna field, which filtering discards (a weak mat might yield 1 raw read per tag vs. 15 for a strong one, but both show up identically in the filtered file). Plan is to parse both: filtered as the primary source for crossing-event metrics, raw specifically for the positioning/reliability diagnostics.

**Release process:** version lives in `pyproject.toml`'s `version` field (source of truth, bumped by hand — no setuptools_scm/git-tag-derived versioning). `CHANGELOG.md` follows Keep a Changelog: an `## [Unreleased]` section at the top gets a bullet added as notable changes happen (don't batch this up at release time), and on release that section is renamed to `## [X.Y.Z] - YYYY-MM-DD` and a fresh empty `## [Unreleased]` added above it. Tag the release commit `vX.Y.Z` in git.

`parsers/bib_chip.py` — `parse_bib_chip(file_path) -> pd.DataFrame` (columns: bib, tag_id; tag_id lowercased to match raw-read case). `parsers/tm_data.py` — `parse_tm_data(file_path, race_date=None) -> pd.DataFrame` (columns: seq, bib, finish_time; pass `race_date` to get a full timestamp instead of a bare time-of-day). `parsers/barcode_scanner.py` — `parse_barcode_scans(file_path, start=None, end=None) -> pd.DataFrame` (columns: bib, scan_time; parses the "barcode scanner data received: N" line, `start`/`end` narrow to the race window since the log spans months). All three have a `sanity_check(df)` and validated cleanly against the Indy 5000 data (499 bib/tag rows, 110 tm-data finishers, 112 scans/110 unique bibs with 2 real rescans, time ranges consistent with each other).

`parsers/backup_time_selected.py` — `parse_backup_time_selected(file_path, race_date=None) -> pd.DataFrame` (columns: bib, name, gender, age, scored_event, clock_start_time, chip_start_time, finish_time, chip_time_elapsed, clock_time_elapsed, backup_time_selected). Parses RDS's "Backup Time Selected" Data Check report (row 1 is a title row, row 2 is the real header — `skiprows=1`). This report shows *any* non-main-stream time selected (TM or Trident File backup, not distinguishable from the report alone) — see the known-limitation note below for why it's needed as ground truth. Validated against Indy 5000: 5/5 rows, bibs [576, 591, 598, 606, 630], all backup_time_selected=True.

`analysis/start_finish.py` — `classify_start_finish(reads, bib_chip, gun_time, gap_factor, start_offset=2min) -> pd.DataFrame` (one row per bib: start_time, start_n_reads, finish_time, finish_n_reads) implements the RDS start/finish theory described below, plus `missed_start_reads(classified)` / `missed_finish_reads(classified)`. `GAP_FACTOR_BY_DISTANCE` holds the per-distance Gap Factor (5K/10K/10M). Validated against Indy 5000: reproduces the 1 missed start (641) and 3 missed finish (598/606/630) exactly, finish-time diff vs. tm-data.csv median +0.42s (matches Trident's ~0.5s BS settle) — but see the known limitation above (misses 2 of the 5 real backup-used bibs).

`analysis/timing_utils.py` — `rolling_count(times, window, include_self=False) -> list[int]`, a shared rolling-window counter used by both `queue_lag.py` (finisher density preceding each scan, for queue detection) and `finish_density.py` (density at/around each finish, for finish-line congestion) — don't reimplement this inline in a third module.

**Finish-line density (`analysis/finish_density.py`):** separate question from queue_lag.py -- congestion *at the mat* vs. congestion *at the scanner*. `finish_density(classified, bin_width=30s)` bins finishers per time bucket; `rolling_finish_rate(classified, window=60s)` gives density centered on each finisher; `finish_gaps(classified)` gives the raw inter-finisher gap distribution. On the Indy 5000: mat_id-agnostic per the theory above, 107 finishers/38.9min (avg 2.75/min), but bursty -- tightest gaps between consecutive finishers were 0.16-0.37s (near-simultaneous), and a real cluster of 7 finishers hit within one 30s bin (~08:49:00-08:49:30, 9 within a 60s rolling window). That burst did *not* show up as elevated bib-scan lag in queue_lag.py's analysis of the same window -- suggests the walk distance to the scanner's position acted as a natural buffer that dispersed the cluster before it reached the scanner. Answers Lou's original question (can we tell a finish-line line forming apart from scanner-position lag): yes, by comparing finish_density's bursts against queue_lag's excess-lag/density correlation for the same time range.

**Is 60s the right window for "how busy"?** Cross-checked against RRTC's finish-line theory (rrtc.net/finishline/theory.html, race-size-agnostic engineering reference, not FSRC-specific): their peak-arrival formula `P = 0.6·N/D` (N=finishers, D=distance km) is expressed in runners-per-minute, so a 60s window matches the field's own standard unit -- not an arbitrary choice. But their formula is calibrated on much larger races (their worked examples are 1,200-3,000 finishers) and overshoots for FSRC's typical field size (100-150, occasionally up to 300): applied to Indy 5000 (N≈110, D=5) it predicts P≈13.2/min, vs. our actual measured peak of 9 in a 60s window. Small local races cluster in short bursts (friends finishing together) rather than sustaining a competitive-race-style peak. Lou's call: add a shorter complementary window for burst detection, skip building the RRTC-formula comparison module (`theory.py`) for now -- the theoretical numbers aren't clearly useful yet, revisit if that changes.

`analysis/finish_density.py` now exposes `STANDARD_WINDOW` (60s) and `BURST_WINDOW` (15s) as named constants; `rolling_finish_rate()` defaults to `STANDARD_WINDOW` but takes either. `report/build.py` computes both into `RaceReport.density_rolling` (60s) and `density_rolling_burst` (15s); both reports and the running summary show both peaks. On Indy 5000 the 15s burst window found something the 60s window's own peak number didn't make obvious: 5 finishers within 15s (~08:49:20) versus 9 within the surrounding 60s -- confirming the burst window catches genuinely sharper clustering, not just a fraction of the 60s peak. `render.py`'s `_render_finish_density(report, verbose)` is shared between detailed/sanitized (same pattern as `_render_read_reliability` — write shared render logic once, don't duplicate inline, since that's exactly what caused the earlier drop/network-interruption wording bug).

**Finish-to-bib-scan queue lag (`analysis/queue_lag.py`):** raw lag (scan_time - finish_time) conflates the barcode-scanner operator's fixed standing distance from the finish (varies by race/operator choice — at Indy the operator stood well back) with actual queue-wait time. `add_excess_lag(lag, baseline_quantile=0.05)` treats a low percentile of observed lag as the walk-only baseline and subtracts it out; `add_finish_rate(lag, window)` adds a rolling count of finishers in the preceding window, to check whether excess lag tracks local finisher density (real queue) or not (just per-scan/walk variability). On the Indy 5000: raw lag median 6.9s/90th-pct 9.65s, excess lag median 3.2s/90th-pct 5.9s, but excess lag vs. 60s-window finisher density correlation ≈ 0 (-0.005) and peak density was only 8 finishers/min for a 110-person field — **no evidence of a real queue forming**, the ~3s excess is more likely per-scan or per-finisher walk-path variability than backlog. `lag_histogram()` gives a quick text-based bin count (no plotting dependency yet).

## Per-race report (`report/`)

`report/build.py` — `build_race_report(race_dir, race_name, race_date, distance, gun_time, notes=None, n_registered=None, n_starters=None, n_finishers=None) -> RaceReport` orchestrates all the parsers/analysis above for one race into a single dataclass. Auto-locates `f_*.log` and the backup report within `race_dir`, and the shared scanner log at `race_dir.parent / "logs" / "barcode-scanner.log"`.

**Participant universe is the union of `tm-data.csv` bibs and any bib with a chip read, not just `tm-data.csv` alone.** Filtering to `tm-data.csv` bibs only silently drops anyone who started (has a chip read) but never got a finish OR a backup entry — found on Indy: bib 658 had a clean start read but zero finish reads and no `tm-data.csv` row at all, so it was invisible in every count until this was fixed. Always check for this class of "orphan" bib (has reads, absent from tm-data) before trusting a race's totals.

**Resolved — Indy bib 658 was a legitimate drop, not a data problem.** Confirmed by Lou: she was seen at the finish line but there's no record of her actually finishing, and RDS has her recorded as a "drop" (started, didn't finish). This is a real DNF, not a missed chip read or backup-coverage gap. Since `zero_reads`/`missed_finish_only` can't tell "legitimate DNF" apart from "technical failure that needs chasing" from data alone (both look identical: no finish read, no backup entry), added `RaceReport.known_drops: list[int]` (manual override, same pattern as `n_registered`/`n_starters`/`n_finishers`).

**A confirmed drop is excluded from the finish-miss count/percentage entirely, not just reclassified within it.** First cut had `known_drops` bibs counted as a subset of "unrecovered finish misses" (reported separately but still part of the headline "N of participants chips were not read at finish" and `pct_read_opportunities_missed()`). Lou pointed out that's wrong: a drop was never going to have a finish read — there's no missed opportunity to count, since the runner didn't reach the finish. `RaceReport.genuine_finish_miss_bibs()` = `finish_miss_bibs() - known_drops` is now what the headline count/percentage and `render_summary_row()`'s "Chips Not Read at Finish" column use; `pct_read_opportunities_missed()` likewise subtracts a drop bib's finish-opportunity from the numerator. `_reconcile_backup()` in render.py works from this genuine set for the recovered/unrecovered split, and reports drops as a fully separate bucket (`n_drops`/`drop_bibs`) rather than nested inside "unrecovered" — on Indy this dropped the headline from 4 to 3 chips not read at finish (bib 658 no longer counted).

**`n_starters`/`n_finishers` are manual overrides, not computed** — there's no data source in this repo for actual registration-system counts (RunSignUp export etc.), and `n_bib_chip_entries` (the `bib-chip.csv` row count) is just the print-run/assignment sheet size, not registrants — don't confuse the two. On Indy: our own computed `n_participants` (data-confirmed) was 111, vs. Lou's authoritative 141 registered / 110 starters / 109 finishers — the gap between 111 and 110/109 wasn't fully reconciled (bib 658 accounts for part of it but not all); pass known-good numbers per race when Lou has them.

**`n_registered` (2026-07-13) is auto-derived, not a manual override, when the rdgo export (zip) path is used** — `rdgo_export.n_registered(export)` counts `entity_map` entries restricted to the race's own scored event(s) (same `_in_scored_event()` scoping as `bib_chip()`/`known_drops()`, so a companion division sharing the same export doesn't inflate the count). Unlike `n_starters`/`n_finishers`, RDS already has this number recorded, so there's no need to ask Lou for it on every race — `build_race_report()` only falls back to a passed-in value (or `None` on the CSV-only path, which has no export to derive it from) when explicitly overridden. Validated on Panda 5K: derives to 300, matching the value Lou had previously supplied by hand.

**Missed-read breakdown is 4-way, not just "missed start"/"missed finish"** — the categories mean different things operationally and shouldn't be conflated:
- `no_chip_assigned` — bib ran the race but has no row in `bib-chip.csv` at all. Real scenario Lou's seen: bibs handed out without a chip attached. Not visible as a "missed read" in the naive sense since there's no chip to read.
- `zero_reads` — chip assigned, but *no* read anywhere (neither start nor finish). Suggests a non-functioning/damaged chip rather than a one-off positioning miss.
- `missed_start_only` / `missed_finish_only` — has a read at the other location but not this one; the "genuine miss" cases validated earlier (598/606/630/658 finish, 641 start on the Indy 5000).

None of these categories auto-classify as a possible equipment/cabling problem vs. a normal one-off from the filtered-file counts alone — that judgment needs the per-mat raw-read diagnostics, built in `analysis/mat_reliability.py`.

**`analysis/mat_reliability.py` — per-mat raw-read diagnostics (detailed report only).** Uses `r_*.log` (RR, every individual antenna ping), not `f_*.log` — filtering collapses exactly the density/signal-strength signal this needs (a weak mat might yield 1 raw read per tag vs. 15 for a strong one, but both look identical once reduced to a single BS/LS record). `mat_summary(raw_reads)` gives per-mat n_reads/unique-tags/reads-per-tag/mean signal strength (RSSI).

**Design corrected once already, per Lou (race director, knows the physical setup):** the first version compared each *missed-read bib's* raw pings against their own filtered-file miss, reasoning that "zero raw reads anywhere" would mean an isolated chip/positioning failure vs. a mat-wide problem. Lou pointed out this doesn't actually work: if a bib has no BS/LS record in the filtered file, they'll essentially always have zero raw reads too — Trident's own onboard filtering silently dropping a *good* raw read would mean a device software bug, which Lou considers vanishingly unlikely. So that per-bib comparison mostly just re-confirms "yes, it's a real miss" and says nothing about equipment health. **The only information raw reads actually carry is per-mat: is a given antenna reading weakly (low `signal_strength`/RSSI) or being missed almost entirely (near-zero reads) — the two symptoms a bad cable/connector would produce.** The per-bib check was removed; `mat_summary`/`flag_weak_mats`/`mat_reliability_verdict` operate purely on per-mat aggregates now.

`flag_weak_mats(summary, drop_ratio=0.5)` flags mats whose unique-tag coverage OR mean signal strength is well below (under `drop_ratio` of) the average of their **immediate neighbors** (`mat_id` ± 1) — not the race-wide median. This matters because, per Lou, read counts/coverage naturally taper from the middle of the mat run toward the edges (fewer runners take the outermost lanes), so a flat race-wide threshold would misflag a legitimately weaker edge mat as a problem. A real cabling/power issue instead shows up as a *local* dip that breaks an otherwise-smooth taper — comparing each mat only to its neighbors catches that regardless of which end of the run is nearer the reader cabling (mat_id 1 isn't assumed to be either edge or middle — Lou himself isn't sure which direction the numbering runs, so the code doesn't assume it). `mat_reliability_verdict(summary, weak_mats)` rolls it up into `{n_mats, weak_mats, systemic_signal}`.

Validated against the Indy 5000 (8 mats): mean signal strength ranged 173-184 and unique-tag coverage 52-109 across mats, all within neighbor-relative tolerance — `systemic_signal=False`, no mat flagged. `report/render.py`'s `_render_mat_diagnostics()` renders this de-emphasized to a single reassuring line in the common case, independent of whether the race had any missed reads at all (it's a general equipment-health check, not tied to specific misses); the per-mat table (including a "weak" flag per mat) and a "possible cabling/power issue" callout only appear when `systemic_signal` is true, per Lou's call that this is a "worth checking" caveat, not a headline metric. Detailed report only — never sanitized (mat-level equipment detail isn't client-facing).

`RaceReport.finish_miss_bibs()` = `zero_reads ∪ missed_finish_only`, the raw technical set (includes confirmed drops). `RaceReport.genuine_finish_miss_bibs()` = that minus `known_drops` — **use this one**, not `finish_miss_bibs()`, for any headline miss count/percentage (the "chips not read at finish" number Lou wants prioritized over raw "recovered" counts). `RaceReport.pct_read_opportunities_missed()` also excludes a confirmed drop's finish-opportunity from the numerator. **`_reconcile_backup()` in render.py splits the genuine misses into recovered-via-backup vs. still-unexplained/needs-review, with confirmed drops reported as a fully separate, non-nested bucket** — don't just report the total backup-report row count as "how many finish misses were recovered," since that total also includes network-interruption cases (591/576-style) that aren't finish misses in our own data at all; mixing the two produces nonsensical arithmetic (caught and fixed once already — recovered + unrecovered must equal the *genuine* finish-miss total, with drops entirely outside that accounting).

**Don't depend on `backup-time-selected.csv`/`time-machine-used.csv` being supplied, and don't trust its own `finish_time` field even when it is.** Lou normally won't provide this report (it's optional, per Data layout below) — `RaceReport.backup` is `None` in the typical case and every render path already handles that. Separately, when it *is* supplied, its `finish_time` only has tenths precision (vs. `tm-data.csv`'s hundredths) and reads as a data mismatch even though it's the same crossing (differences were 0.01-0.09s on Indy, i.e. noise, not a real discrepancy) — so `RaceReport.tm_data` (always available) is the source used to display `finish_time` in the "Backup timing used" table; the backup report itself is used only to know which bibs used backup and to look up their name.

**All displayed timestamps are formatted to hundredths via `render.py`'s `_fmt_ts()`**, not Python's default `str(pd.Timestamp)` (which shows meaningless microsecond zeros, e.g. `08:49:19.300000` instead of `08:49:19.30`) — use `_fmt_ts()` for any new timestamp display rather than interpolating a Timestamp directly into an f-string. `_fmt_time(ts)` is the date-free counterpart (hundredths, time-of-day only) for a timestamp displayed alongside a date already shown separately — e.g. the detailed report's Overview line uses it for `gun_time` rather than `_fmt_ts()`, since `race_date` is already in the same line (found 2026-07-13: the Overview line had been interpolating `r.gun_time` directly, the exact anti-pattern this note already warned against, showing raw microseconds).

`report/render.py` — three outputs, all markdown:
- `render_detailed_report()` — bib-level detail (names, bib numbers, individual times, race name, full queue-lag section). **Never publish this outside the repo.**
- `render_sanitized_report()` — aggregate-only (counts/percentages/distributions, no bibs, no names, **no race name** — Lou doesn't want client identity exposed on a public wiki page), per Lou's requirement for anything headed to fsrc-tech. Missed reads are framed as "recovered results" language. **No queue-lag section** (Lou's call — kept in the detailed report only, still fully available via `queue_lag.py` for ad hoc investigation). Opens with a YAML front-matter block (`title: <race_date>`) ahead of the `# Race Timing Summary` H1 — MkDocs reads `title` from front matter in preference to the first heading, so the wiki nav shows the race date (needed once there are many of these) while the page body keeps the more descriptive heading.
- Both detailed and sanitized reports include a "Notable Occurrences" section from `RaceReport.notes` (free text, human-supplied — e.g. "brief network outage, did not affect results"). Safe to include even in the sanitized version since it's about race operations, not individuals.
- **`_render_read_reliability(report, include_bibs)` is shared by both reports** (detailed passes `True`, sanitized `False`) specifically to prevent the two renderers' wording/grouping from drifting apart — caught once already: an earlier version indented the "N confirmed as a drop" line right under the network-interruption bullet, implying the drop was one of the network-interruption cases, when they're actually disjoint groups. Confirmed drops are now reported as their own separate line entirely (not nested under recovered/unrecovered at all — see `genuine_finish_miss_bibs()` above). Don't duplicate this logic inline in either render function again; extend the shared helper instead.
- `render_summary_row()` / `append_summary_row()` — one markdown table row per race (also no race name, no queue lag), inserted **sorted by race date descending** (most recent race first) into the running cross-race summary (trend tracking over time, not per-race detail); re-generating an already-listed race replaces its row rather than duplicating it. (Despite the name, `append_summary_row` doesn't literally append or blindly prepend — an earlier version did blindly prepend, which silently broke "most recent first" the moment a race was [re-]generated out of chronological order, e.g. re-running an earlier race's report after a later one was already in the table — caught by Lou on the Parkway Panda 5K (2026-05-25) ending up above the Indy 5000 (2026-07-04).) No dedicated "No Chip" column: a nonzero `no_chip_assigned` count is rare (seen once, from a bib-printing process problem, not a chip-timing issue) so it's folded into the Notes cell (`_summary_notes_cell()`) only when it actually happens, rather than reserved as a permanent mostly-empty column. The Notes cell itself is truncated to 60 chars (`_NOTES_CELL_MAX_LEN`) — mkdocs-material's table rendering blew up vertically wrapping long free text in a narrow column; the full sentence is still on the linked per-race page, so the summary row only needs a teaser. The summary's own H1/title is "Timing Summary" (`_SUMMARY_HEADER`, renamed from "Running Timing Summary" — simpler, and it's what the fsrc-tech reports folder's `README.md` index shows too, see below).

**`reports/SUMMARY-LEGEND.md` (2026-07-13)** — the column-definition paragraph that used to sit directly under `_SUMMARY_HEADER` (dense, one long paragraph ahead of the table) is now its own page, generated by `render_summary_legend()` and linked from `SUMMARY.md`'s one-line intro instead of inlined. `append_summary_row()` (re)writes it on every call, alongside the row it's actually appending, so it can't go stale even though its content doesn't depend on any particular race. Lives next to `SUMMARY.md` in `reports/` (and, after publishing, in fsrc-tech's `docs/race-services/reports/`, same filename both places so the link resolves in both locations) — see Publishing below, `publish.py` now copies three files, not two.

**`RaceReport.notes` vs. `detail_notes` — two different confidentiality levels, not the same thing:** `notes` is shown in ALL THREE outputs (detailed, sanitized, summary) and must stay free of bib numbers, names, and anything that reads as operator/volunteer error — e.g. the network-interruption note is fine (system-level, no blame). `detail_notes` is shown ONLY in the private detailed report and is where bib-specific or operationally-sensitive narrative belongs — e.g. "bib 669 was missing due to a check-in tablet that didn't sync (operator error)" is `detail_notes`, not `notes`, per Lou's explicit call: operator-error-flavored specifics stay internal even though the underlying network-interruption *fact* was fine to share. When in doubt about which list a note belongs in, treat "names a bib/individual" or "assigns human blame" as the bar for `detail_notes`.

**Sanitized report filename is date-only (`reports/<race_date>.md`), not the race slug** — e.g. `reports/2026-07-04.md`, not `reports/2026-07-04-indy-5000.md`. This file is git-tracked (goes into GitHub), and per Lou, the race identity shouldn't be discoverable just from browsing the repo's file listing even though the sanitized content itself was already race-name-free. `reports/SUMMARY.md`'s "Recovered (Backup Used)" column was also removed per Lou's call (kept in the per-race sanitized report's Read Reliability section instead, where the recovered-vs-unrecovered breakdown has room to be explained).

`report/generate.py` — `argparse`-based CLI (`python -m chip_timing_analysis.report.generate <race_dir> [--race-name ...] [--race-date ...] [--distance ...] [--gun-time ...] [--note ... (repeatable)] [--detail-note ... (repeatable)] [--n-registered N] [--n-starters N] [--n-finishers N] [--known-drop BIB (repeatable)] [--exclude-bib BIB (repeatable)] [--reports-dir DIR]`, see `--help`) that writes `<race_dir>/report.md` (detailed, gitignored with the rest of `data/`), `reports/<race_date>.md` (sanitized, git-tracked, copy-ready for fsrc-tech — filename is the race date only, not the slug), and appends a row to `reports/SUMMARY.md` (git-tracked). Every `generate()`/`build_race_report()` parameter is now CLI-exposed (2026-07-12; earlier versions of this CLI were positional-only and didn't expose `n_registered`/`n_starters`/`n_finishers`/`detail_notes`/`known_drops`/`exclude_bibs` — direct-from-Python was the only way to pass them). `race_dir` is the only required positional; `race_name`/`race_date`/`distance` are now optional flags too (2026-07-12) — when `race_dir` has an RDS export zip, any left unset are derived from it: `race_name` from `race["name"]`, `race_date` from `race["start_date"]` (RDS's own configured date, already `"YYYY-MM-DD"` — preferred over deriving from gun_time's date, which needs a timezone conversion and could in principle land on a different calendar date), and `distance` from `rdgo_export.distance_label()` (per-`scored_event`, read directly from the export's `integration_events` — each `scored_event`'s `included_event_ids` can reference more than one `integration_event`, as on Panda 5K where the "Panda 5K" scored_event's two age-based registration categories both agree on distance "5K"; `distance_label()` checks for that agreement rather than assuming one-to-one, and returns `None`, triggering a fallback, if they disagree or the field's missing). **Deliberately not primary: reverse-mapping Gap Factor against `GAP_FACTOR_BY_DISTANCE`** (`build.py`'s `_rdgo_distance_label()`, now the fallback only) — per Lou, Gap Factor (`min_finish_time`) is itself a user-adjustable RDS timing setting, not a fixed distance encoding, so nothing guarantees a race director left it at this codebase's assumed default; a race director who customized it would've silently gotten the wrong distance label under the old gap-factor-only approach. All of the above are required and raise a clear `ValueError` only on the CSV-only fallback path, where there's no export to derive them from. `--gun-time`/`--note`/`--race-name`/`--race-date`/`--distance` all treat a **blank or whitespace-only** string the same as an omitted flag (not a literal value) — not just exact `""` — specifically because `.vscode/tasks.json`'s optional `promptString` inputs default to a single space `" "`, not `""` (see below for why). `generate()` uses the *resolved* `report.race_date` (not the possibly-`None` argument) for the sanitized report's filename, since the argument itself may have been derived inside `build_race_report()`. `render_summary_row()`'s Date cell is a markdown link to that race's `<race_date>.md` — relies on the summary file and the per-race sanitized reports living side by side in the same directory, which holds both here (`reports/`) and after publishing to fsrc-tech (below).

**`.vscode/tasks.json`** — "Chip Timing: Generate Race Report" task runs the CLI above via Command Palette → "Run Task," prompting for `race_dir` (only truly required field) plus `race_name`/`race_date`/`gun_time`/`note` (`promptString`) and `distance` (`pickString`: `"(auto-detect from RDS export)"` default plus 5K/10K/10M). For a zip-based race, hitting Enter through every prompt is enough. Validated at the CLI level (2026-07-12) against Panda 5K (single distance) and Wild Trail 5K & 10K (combined) with everything left blank, and against Indy 5000 (no zip, CSV-only) with everything filled in — all reproduced their previously-validated numbers exactly. Later validated for real (2026-07-13) against the Lewis Memorial 10 Miler (first real 10-mile race through this pipeline) — its `integration_events` distance field is the human-readable `"10 Miles"`, not `"10M"`, confirming this field is display-only in the zip path (Gap Factor there comes from the export's own `min_finish_time`, never a `GAP_FACTOR_BY_DISTANCE[distance]` lookup) and doesn't need to match this codebase's key format.

**Two related VS Code gotchas, found only once Lou actually ran the task from the Command Palette — CLI-level testing (`--race-name ""`, etc.) catches neither, since both are specific to how VS Code itself resolves/assembles task inputs, not to the CLI:**
1. **(found/fixed 2026-07-12)** A `promptString` input whose `default` is exactly `""` and is left untouched can resolve as *cancelled* rather than as an empty string, silently aborting the whole task with no error dialog — hit `raceName`/`raceDate`/`gunTime`/`note` (then `"default": ""`) the first time Lou tried the blank-to-auto-derive flow, right after confirming the `raceName` prompt.
2. **(found/fixed 2026-07-13, one real run after gotcha 1's fix)** Separately, an empty-string *argument value* — whatever its source — can vanish entirely from the shell command line VS Code assembles, rather than being passed as an empty argument; this hit the `distance` picker's `"(auto-detect from RDS export)"` option (`"value": ""`), which collapsed `--distance` and the following `--gun-time` flag together and broke `argparse`. This is a distinct failure mode from gotcha 1 (a `pickString` selection is never "left blank/untouched" the way a `promptString` can be) — so fixing gotcha 1 did not fix this one, and the original version of this note wrongly suggested a `pickString` auto-detect option was a safe pattern.

`race_dir`/`publishRaceDate` (no default at all, genuinely required) don't have either problem — an empty Enter there *should* cancel. **Fix for both: never let an optional input resolve to a literal `""`** — the four `promptString` defaults and the `distance` picker's auto-detect value all use a single space `" "` instead, and `generate.py`'s blank-detection strips whitespace before checking rather than comparing to exact `""` (see above). **Takeaway for extending this file: give every optional input (whether `promptString` default or `pickString` option value) a non-empty sentinel like `" "`, never bare `""` — and re-verify any input-prompt change by actually invoking the task from the Command Palette, not just by simulating the resulting CLI call; both gotchas above only surfaced that way.**

The rarer per-race overrides (`n_registered`/`n_starters`/`n_finishers`, `known_drops`, `exclude_bibs`) are CLI flags but deliberately not wired into the task's prompts — infrequent enough (see the Indy/Panda case studies above) that adding them as always-shown prompts would be more friction than the terminal invocation they replace. Add a second task or extend this one if that trade-off stops being right.

Sanitized output is copy-ready but publishing is a manual step (Lou's call, not automated) into the `fsrc-tech` wiki's `docs/race-services/reports/` folder (sibling of `docs/race-services/timing.md`, the page that explains how this data gets produced) — check that fsrc-tech is locally checked out (see `~/.claude/CLAUDE.md`'s fsrc-tech sync note) before doing this.

1. Run `report/generate.py` for the race as usual (see above) — this refreshes `reports/<race_date>.md` and appends a row to `reports/SUMMARY.md` (and rewrites `reports/SUMMARY-LEGEND.md`, see above) in *this* repo.
2. Copy `reports/<race_date>.md` verbatim to `fsrc-tech/docs/race-services/reports/<race_date>.md` (new file, one per race).
3. Copy `reports/SUMMARY.md` over `fsrc-tech/docs/race-services/reports/README.md` (overwrite, not append/merge — this repo's `SUMMARY.md` already accumulates every race, so the copy fully replaces the fsrc-tech index each time; naming it `README.md` there satisfies fsrc-tech's convention that every `docs/` topic folder has a `README.md` acting as an index, and its table's per-race links resolve correctly since it stays in the same directory as the `<race_date>.md` files).

6. Add a dated entry to `fsrc-tech/CHANGELOG.md` noting the new race report, and commit/push in fsrc-tech (per Lou's own call — not run proactively).

Deliberately stops there and prints a reminder of steps 5-6 rather than doing them — per Lou's explicit call above, committing/pushing in fsrc-tech stays a deliberate manual step, not something run proactively; automating the file copies (which are just mechanical) doesn't change that. ## Data layout

`data/logs/` — shared, long-lived tmtility app logs (`barcode-scanner.log`, `tm-reader.log`), **not** per-race (see above). Often incomplete — don't assume every setup has them.

**`race_dir` (the first argument to `build_race_report()`/`generate()`/the "Chip Timing: Generate Race Report" task) is not required to live under this repo's `data/`** — it can be any path, including a race folder stored entirely outside the repo (e.g. `C:\Users\lking\Dropbox\...\past races\<race>`, confirmed working end-to-end on the Lewis Memorial 10 Miler, 2026-07-13). `generate()` always writes the detailed `report.md` to `<race_dir>/report.md`, wherever that is — so for a race stored outside the repo, the detailed report lands in that external folder too, not in `data/`. Only races whose folder actually lives under `data/<date>-<race-slug>/` (below) show up there; don't assume a race is missing/broken just because it's absent from `data/` — check wherever its `race_dir` actually points first.

Per-race directories under `data/<date>-<race-slug>/` (e.g. `data/2026-07-04-indy-5000/`), flat (no subfolders) — this layout is this repo's own convention for races kept inside it, not a hard requirement of the tooling itself (see above):
- `r_*.log`/`f_*.log`/`s_*.log` — raw Trident read files, directly at race root
- `tm-data.csv` — tmtility data export / RaceDay Scoring input
- `bib-chip.csv` — bib/chip assignment list. A race can draw chips from more than one sheet (e.g. a companion program's own reusable pool covering a different bib range) — check for a second sheet and merge before trusting `no_chip_assigned` (see the Parkway Panda 5K case study below)
- `backup-time-selected.csv` / `time-machine-used.csv` — RDS Data Check report exports, when pulled (optional, ground truth for which bibs used a non-main-stream time — see `parsers/backup_time_selected.py`)
- `results-<race name>.csv` — official RunSignUp results export (optional; columns: Result Id, Place, Bib, First Name, Last Name, Gender, Age, City, State, Country, Clock Time, Chip Time, Pace, Age Percentage). No parser yet (read ad hoc with `pd.read_csv`) — this is the authoritative "who's officially scored" list, useful for reconciling `tm-data.csv` against real starter/finisher counts. RunSignUp's results page itself is JS-rendered (WebFetch on the page URL gets nothing useful); its results are reachable via `https://runsignup.com/Rest/race/<race_id>/results/get-results?format=json&event_id=<event_id>&result_set_id=<result_set_id>&page=N` (race_id/event_id/result_set_id scraped from the results page URL/source — fragile, prefer asking Lou for the CSV export directly, which is more reliable and is what ultimately resolved the case below).

**Resolved case study — Indy bib 669 (highest bib issued):** `tm-data.csv` initially had 110 bibs vs. 109 official results, with 669 the one bib present in `tm-data.csv`/chip reads but missing from official results. Root cause, confirmed by Lou: a **check-in sync failure**, not a missed chip read or a registration failure — check-in was done correctly on a tablet, but the tablet was shut down before syncing to RunSignUp. Re-starting the tablet synced the pending check-in and resolved it; `results-<race>.csv` was re-downloaded and now matches `tm-data.csv` exactly (110/110, zero discrepancies either direction). This is a distinct failure mode from anything in the missed-read breakdown (`no_chip_assigned`/`zero_reads`/`missed_start_only`/`missed_finish_only`) — the chip worked fine throughout, the gap was purely in the check-in-to-registration-system sync. A `not_in_official_results` report category (participant bibs with chip activity but no `results-<race>.csv` row, mirroring `no_chip_assigned`'s pattern against `bib-chip.csv`) was proposed but not built — worth reconsidering if this recurs, but treat as a one-off for now since it resolved cleanly via re-sync rather than needing detection tooling.

**Parkway Panda 5K (2026-05-25) — reconstructed-data race, validated end-to-end.** Lou lost the original `tm-data.csv` to a tmtility laptop DB wipe and rebuilt it from RDS's own per-read "Time Machine" Data Check export (`Panda5K1KFunRun_20260525_timemachine.csv`, kept in the race dir for audit trail but not read by any parser — its per-read `Used`/`Reason` columns don't match `tm_data.py`'s or `backup_time_selected.py`'s expected layout). Cross-checking a few finish times against that export's full ISO timestamps confirmed the reconstruction was accurate; the only defect was ~14 rows with a blank bib (unresolved scanner reads, e.g. "Unknown Bib or chip") that needed dropping before `parse_tm_data` (which enforces `bib` as non-null int64) would parse the file — a generally-applicable checkpoint if `tm-data.csv` is ever hand-reconstructed from this kind of raw per-read report again.

No gun_time was supplied up front, so it was derived directly from the raw log's own `ab` GUNTIME marker records (per the format spec above) — 2026-05-25 09:00:49.46 — and cross-validated two independent ways: (1) `Chip Time`/`Clock Time` arithmetic in the (malformed, see below) backup report is internally consistent only with that gun time, and (2) it matches the RDS per-read export's real timestamps exactly. Worth remembering as a general technique when gun_time isn't otherwise given: derive it from the raw log rather than asking, then sanity-check against any other available report.

Lou's hand-typed `time-machine-used.csv` for this race had its "Time of Day" columns truncated to `MM:SS.t` (missing the leading `HH:` hour, e.g. `00:49.4` instead of `09:00:49.4`) — confirmed via the gun-time cross-check above and safe to repair by prepending the known hour since the whole race stayed within one clock hour. `parsers/backup_time_selected.py` was also missing a real robustness gap this race exposed for the first time: "Chip Start Time of Day" is blank for a bib with no chip assigned at all (nothing to report), which used to crash the whole parse — fixed to parse as `NaT` (see CHANGELOG). Two lessons for reruns: check whether a "Time of Day" export has plausible full-hour values before trusting it at face value, and don't assume every row in a Data Check report has every column populated.

This race also runs a companion **"Low Key"** program sharing the same timing infrastructure, with its own separate, mostly-static bib/chip assignment sheet (`bib-chip assignments - Low Key 501-999 (2026).csv`, bib range 501-999, dated months before the race — a reusable weekly-series sheet, not race-specific) that Lou hadn't initially included. 10 of this race's participants (bibs 545-554) drew from that pool but were scored as "Panda 5K" like everyone else (confirmed via RDS's own `Scored Event` column in the per-read export) — same distance/gap-factor, no special handling needed once their chip mapping was merged in. **Always check for a second bib-chip sheet covering an out-of-range bib block before concluding a batch of bibs has "no chip assigned"** — the two failure modes (genuinely no chip vs. chip assigned via a sheet you don't have yet) look identical from `no_chip_assigned` alone. After merging, this race's genuine `no_chip_assigned` count is 12 (bibs 377-386, 388, 389 — a real batch handed out without a chip attached, matching what Lou described going in), not the 22 an incomplete bib-chip.csv would suggest.

**Bib 375's chip cross-reference in RDS was a data-entry error, not a real chip issue: a test/calibration chip that was never actually issued to that participant.** Went through two wrong framings before landing on this, worth remembering the progression since each correction came from Lou, not from re-examining the data: (1) first pass called it a "staff/test bib" to exclude entirely, which conflated the chip with the bib/participant; (2) second pass called it a real participant with a non-functioning chip (`zero_reads` — "possible non-functioning chip"), reasonable given its data (~339 pre-gun bench-test reads 08:02:31-08:03:22, one post-gun Time-Machine bib-scan at 09:01:52, zero real chip reads) but still wrong; (3) Lou's actual correction: the chip was simply never given to the participant at all, so this bib should be **excluded from the participant universe entirely**, not bucketed into any miss category. Handled via a new `exclude_bibs` parameter on `build_race_report()`/`generate()` (distinct from `known_drops`: full exclusion from every count, not just miss-recategorization — for a data-entry error, not a real DNF). General takeaway: when a chip's reads look physically implausible (a burst far from any real crossing, like this one 58 minutes before gun), don't assume "malfunctioning" — ask whether the assignment itself might be wrong.

Validated end-to-end (5K, 774 bib/chip entries after the Low Key merge, 261 confirmed participants with bib 375 excluded): 0 zero-reads, 5 missed-start-only, 6 missed-finish-only, of which 3 (148, 277, 331) were recovered via backup timing. The other 3 (267, 303, 333 — no chip finish, no backup finish, no tm-data.csv row at all) were the same open-question shape as Indy's bib 658; **Lou confirmed these are drops** (`known_drops=[267, 303, 333]`), dropping the headline to 3 genuine finish misses (1.1%). No mat reliability issues flagged (8 mats, all within neighbor-relative tolerance).

**`pct_read_opportunities_missed()`'s definition changed (2026-07-10), per Lou:** a no-chip-assigned bib no longer counts as "2 missed opportunities" in the numerator — there was no chip, so there was no chip-read opportunity to miss in the first place (distinct from `zero_reads`, where a chip *was* assigned and genuinely never worked, which still counts as 2 missed). The denominator (`n_participants * 2`) is unchanged — a no-chip bib is still a real participant. On Panda this dropped the headline from 6.1% to 1.5%; Indy is unaffected (it never had a no-chip bib, so the removed term was always 0 there).

**Found and fixed a real render bug this race exposed for the first time:** `_reconcile_backup()` in `render.py` labeled *any* backup-used bib that wasn't a finish-miss as a "network interruption" case (Indy's 576/591 pattern) — but a no-chip-assigned bib is *also* backup-used and *also* not a finish-miss, for a completely different, already-reported reason. Indy never had a no-chip bib in its backup report, so this never surfaced there. Fixed in both the aggregate reconciliation and the per-bib "Backup timing used" table's reason column (now says "no chip assigned" instead of misattributing it to a network drop) — see CHANGELOG.

**Found a real classification bug via Lou's own read of the report, not data validation:** of the 8 original `missed_start_only` bibs, 3 (180, 314, 349) actually had a clean LS start read 0.04-0.30s *before* `gun_time` — front-of-field runners whose chip crossed the mats a fraction of a second before the automatic GUNTIME marker registered. The old `[gun_time, gun_time + start_offset]` window (no lower slack) excluded these as pre-gun noise, the same rule that correctly excludes genuine bench-test reads (e.g. bib 375's chip, ~58 minutes before gun — see above). `classify_start_finish()` now takes `pre_gun_grace` (default 1s, `DEFAULT_PRE_GUN_GRACE`) and starts the window at `gun_time - pre_gun_grace` instead — small enough to not reach anywhere near real pre-race test noise. The remaining 5 missed-start bibs (110, 172, 173, 327, 341) are genuine: scattered across ~09:22-09:54 with no shared mat or time cluster (bib 110 also had a burst of 126 reads 34 minutes before gun with nothing between that and its finish — likely idle time near the mats, not a start crossing) — **no evidence of a systemic missed-start issue** on this race, once the false positives are removed. Reproduces Indy's validated 1 missed start (641) unchanged.

**RDS's own full database export — now the normal ingestion path in `build_race_report()` (2026-07-10), when a zip is present in `race_dir`.** Lou added `Panda 5K & 1K Fun Run.zip` to the race dir — not a "Data Check" report but a full export of RDS's underlying database, bundled with ~40MB of unrelated app diagnostic logs Lou confirmed can't be separated out at export time. Full scoping writeup: `~/.claude/plans/graceful-honking-sky.md` (2026-07-10). Two JSON members inside it matter:
- `<race name>.json` — race/participant config: `chip_cross_ref` (875 rows, the pre-printed bib-chip assignment sheet, same role as `bib-chip.csv` — includes unused spares, not the participant-truth source), `participants`/`devices`/`scored_events`/`timing_locations`.
- **`filterer.data`** (a second JSON file, easy to mistake for a log next to `main.log`/`api.log`) — RDS's own already-computed per-read scoring decision, not raw reads needing reclassification: `entity_map` (bib/chip/drop-flags per participant — this, not `chip_cross_ref`, is the correct bib-chip source, since it reflects actual participant reality including a no-chip bib as an empty `chips` list) and `read_data` (per scoring_entity_id: `occurrence` 1=start/2=finish, `device_id` of the stream that produced it, `timeshift_timestamp`). A genuine miss is simply the *absence* of the other occurrence — verified clean on every case checked (finish misses 267/303/333, start misses 110/172/173/327/341, no-chip bib 381).

`parsers/rdgo_export.py` implements this: `parse_rdgo_export(zip_path) -> RdgoExport`, plus `gun_time()`/`gap_factor()`/`bib_chip()`/`known_drops()`/`classify_start_finish()` producing the same DataFrame shapes the CSV-based pipeline already consumes (so `missed_start_reads()`/`missed_finish_reads()` from `analysis/start_finish.py` work unchanged on its output). **Don't hardcode which device_id means "chip" vs "backup"** — device_id assignment is per-installation (Panda 5K happened to have 2=Trident/1=Time Machine, nothing guarantees that numbering elsewhere); resolve it from the race's own `devices` list via `is_chip_device()`/`device_label()` instead (a live antenna reader has a `hardware_name`, a file-fed device doesn't) — caught by Lou during review, the first draft had this reasoning only in a docstring, not actually resolved from data.

Full validation against Panda 5K (comparison script was throwaway/scratchpad, not committed, per the plan): gun_time and gap_factor matched exactly; `no_chip_assigned` matched exactly (12 bibs); every common finish_time matched to the millisecond (0.0s diff across 243 bibs); the "recovered via backup" set (148/277/331) matched exactly once resolved via `is_chip_device()` instead of raw device_id. Two things needed correcting before they matched: (1) `known_drops` needs `rdgo_dnf OR rdgo_drop`, not `rdgo_drop` alone — checked against Panda's 3 confirmed drops, `rdgo_drop` was `False` on all three; Lou clarified she'd marked them DNF instead, and explained why: "drop" removes a participant from RDS's in-progress board, DNF doesn't, and which one she uses isn't a consistent choice, don't assume either alone is complete. (2) Bib 375 (a real participant whose chip was a test/calibration chip accidentally assigned instead of a proper race one — see below) shows up with a real post-gun Time-Machine bib-scan but zero qualifying chip reads at all; RDS's own `rdgo_dnf` flag is `True` for it too, so applying `known_drops` the same way naturally excludes it, same as the 3 real drops.

**Schema-stability risk, addressed:** Lou asked RaceDay Scoring support directly (Support Incident #3613351, 2026-07-10). Response from their Product Manager: the zip export is an officially supported mechanism (how RDS moves races between installations and versions, including importing an older backup into a newer version) — not an undocumented internal-only format. Field names are stable in practice; new fields may appear as features are added, so a parser built against this should tolerate unknown fields rather than assume the schema is closed. `parsers/rdgo_export.py` currently reads only the specific keys it needs (naturally tolerant of unrelated additions) but doesn't yet validate/warn on missing *expected* keys.

**Integration into `build_race_report()`:** `build.py` looks for a `*.zip` in `race_dir` first; if found, it supersedes `bib-chip.csv`/`tm-data.csv`/`backup-time-selected.csv`/`time-machine-used.csv` entirely (none of those are read), and `gun_time` becomes optional (read from the export unless explicitly passed — `distance` is still required, for display and as the CSV-fallback path's `GAP_FACTOR_BY_DISTANCE` key). Falls back to the original CSV-based ingestion when no zip is present (e.g. Indy 5000, predates this export existing) — re-verified unchanged after this integration (1 missed start, 3 genuine finish misses, 1.8% missed, exactly as before). `known_drops` passed by the caller is unioned with RDS's own `rdgo_dnf`/`rdgo_drop` flags rather than replacing it.

Four real bugs surfaced only once real integration (not just ad hoc comparison) forced every code path to actually run, all fixed:
- **`entity_map` covers every registrant under the race umbrella, not just the one scored_event `build_race_report()` targets.** `rdgo_export.bib_chip()`/`known_drops()`/`classify_start_finish()` now restrict to participants whose `event_id` is in `scored_event["included_event_ids"]` (`_in_scored_event()`) — without it, a companion division sharing the same export (there wasn't one that mattered on Panda, but the field itself had 433 registrants vs. 300 actually in the "Panda 5K" scored event) would leak into every count.
- **A backup-recovered finish isn't a "miss" from `read_data`'s point of view, but `missed_finish_only`/`missed_start_only` need to mean "the chip specifically missed it"** (the only thing they could ever mean on the CSV path, which only ever sees chip data). `read_data`'s `occurrence` is present as long as *either* stream provided a value, so bibs 148/277/331 (chip missed the finish, Time Machine correctly substituted) don't show as a "miss" in rdgo's own view — first pass at the integration let this render as "network interruption" (wrong; the Indy 576/591 pattern is a different failure mode) instead of "recovered via backup" (right). Fixed by building a `chip_only_classified` view in `build.py` (start/finish nulled to `NaT` whenever `is_chip_device()` says the source wasn't a chip) used only for miss-*detection*; the real, backup-inclusive times from `classified` are still what's rendered.
- **`rdgo_export.known_drops()` originally used `rdgo_dnf OR rdgo_drop` with no other filter, which is far too broad** — RDS sets `rdgo_dnf` on every non-finisher as routine post-race bookkeeping, including no-shows who never started at all. Caught by Lou reviewing the raw list (~42 bibs, most unrecognizable as real drops). Checked directly: the 3 confirmed mid-race drops (267/303/333) and bib 375 (see below) all have `rdgo_dns=False` (they started); the ~38 others all have `rdgo_dns=True` (never started). Fixed by adding `not rdgo_dns` to the filter — `known_drops` in this codebase specifically means "started, didn't finish" (see `RaceReport.known_drops`'s own docstring), not "RDS marked DNF for any reason."
- **RDS's own occurrence data has no equivalent to `pre_gun_grace`** (`analysis/start_finish.py`'s fix for front-of-field runners whose chip crosses a fraction of a second before the automatic GUNTIME marker) — `read_data` simply lacks occurrence 1 for those bibs, indistinguishable in shape from a genuine missed start, so wiring up the rdgo path as primary silently regressed that already-fixed bug (bibs 180/314/349 reappeared as "missed start"). Fixed in `build.py`: before treating a missing rdgo start as real, check whether the CSV-based classification (which does apply `pre_gun_grace`) recovers a start *before* gun_time specifically for that bib, and backfill it if so. A local read at a normal *post*-gun time that RDS's record still lacks is a different, more concerning question — left to `network_interruption_suspects` (next paragraph), not silently patched here.

All four fixes together restore the exact previously-validated Panda 5K numbers: 12 no-chip, 1 zero-read (375), 5 missed-start-only, 6 missed-finish-only (3 recovered via backup: 148/277/331), 4 confirmed drops (267/303/333/375), 6.1% missed.

**`RaceReport.network_interruption_suspects` (new field):** when both a zip and `f_*.log` are present, cross-checks every *remaining* rdgo-reported miss (after the pre_gun_grace backfill above) against the local Trident log — `read_data` only reflects what reached RDS live, so a chip read can exist on the device and never arrive there (Indy's 576/591 pattern, discovered before this repo had the rdgo path at all). Added per Lou's point that raised this ("the Trident logs are a good backup, e.g. in case of network interruption"). Empty on Panda 5K once the pre_gun_grace backfill is applied first — the field is specifically for a genuine post-gun discrepancy the export alone can't explain, not a catch-all. Not yet surfaced in `render.py`.

Next steps if pursued further: re-validate `rdgo_dnf`-as-known-drops (with the `rdgo_dns` filter) and the event-scope filter against a second race before trusting either unconditionally, and decide whether/how to surface `network_interruption_suspects` in the rendered reports.

**Combined multi-distance races (e.g. Wild Trail 5K & 10K, 2026-04-04) — one zip, two RDS "scored events" sharing a single timing setup.** First race exercising this: same mass start/gun time and one shared `timing_location` for both distances, but each distance is scored as its own RDS scored_event with its own Gap Factor (`min_finish_time`: 840000ms/14min for the 5K, 1800000ms/30min for the 10K) — confirmed by inspecting the zip's `<race name>.json` directly, not assumed. This exposed a latent bug that Indy/Panda's single-distance data had never triggered: `gap_factor()` had been reading `timing_location.gap_factors[0]` (one shared value, coincidentally equal to the 5K's `min_finish_time` on both prior races) rather than each scored_event's own `min_finish_time` — silently wrong the moment two distances with different Gap Factors share one timing_location. Per Lou's call, both distances render as **one combined report** (not two separate per-distance reports/summary rows) — matches the existing one-race-dir-one-report file layout and avoids inventing new filename/summary-row conventions for what's still operationally one race.

Implementation: `RdgoExport.scored_events` (plural) replaces the old single `scored_event`; `_event_for_participant()` resolves which of the export's scored events a given entity_map participant belongs to (by `event_id` membership in that event's `included_event_ids`), and `_in_scored_event()`/`bib_chip()`/`known_drops()`/`classify_start_finish()` all use it so a bib from *either* distance is included. `bib_gap_factors()` returns a per-bib `{bib: Timedelta}` mapping (each bib's own scored_event's `min_finish_time`) instead of a single `gap_factor()` value. `gun_time()` still assumes one shared gun time across all scored events (true here — both events' `actual_start_ts` matched exactly) but now raises rather than silently guessing if that ever isn't true (a staggered-start race would need real handling, not this shortcut). `build.py`'s two CSV-based cross-checks that need a gap factor (the `pre_gun_grace` backfill and `network_interruption_suspects`) now run per gap-factor group via `_csv_classify_grouped()` (one `analysis.start_finish.classify_start_finish()` call per distinct gap factor present, concatenated) instead of applying one shared value to every bib — since RDS's own `read_data` occurrence classification (the primary source, unaffected by any of this) doesn't need a gap factor at all, only these two local-log cross-checks do.

Validated: re-ran Indy 5000 and Panda 5K after the refactor and both reproduce their exact previously-validated numbers unchanged (Indy: 111 participants, 1 missed start, 3 genuine finish misses, 1.8% missed; Panda: 261 participants, 5 missed-start-only, 6 missed-finish-only with 3 recovered via backup, 4 confirmed drops, 1.5% missed) — confirming the single-shared-value bug never actually affected either single-distance race's output. Wild Trail 5K & 10K itself: 87 participants, 0 no-chip/zero-reads, 1 missed-start (519), 1 finish-miss recovered via backup (510), no weak mats, no `network_interruption_suspects` — a clean small race, no manual `known_drops`/`exclude_bibs` needed yet.
